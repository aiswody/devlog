## 무엇을 하려 했나

부동산 사무실에서 계약서를 처음 받아보는 순간은 대부분 **계약 당일, 서명 직전**이다.
특약사항 예닐곱 줄을 읽고 판단할 시간은 몇 분뿐이고, 옆에서 중개사는 "다 표준
문구예요"라고 말한다. "원상복구", "근저당", "제소전화해" 같은 말이 뭘 의미하는지,
뭐가 위험한지 모르는 채로 서명하게 된다.

기존 서비스를 조사해 보니 공백이 명확했다. 내집스캔·집체크류는 계약 **전** 매물과
임대인을 검증하고, 로폼 같은 계약 검토 서비스는 기업 법무팀용이다. **계약 당일,
눈앞의 계약서 문서 자체**를 조항 단위로 분석해 주는 도구는 없었다. 그걸 만들기로 했다.

## 무엇을 만들었나

```
계약서 촬영 → OCR → 사용자 교정 → 조항 자동 분리
→ 조항별 위험도(3단계)·카테고리(7종) 분류
→ 판정 근거 신호 표시 → 요약 리포트
```

사진을 찍으면 CLOVA OCR로 텍스트를 뽑고, 사용자가 오인식을 교정한 뒤, 조항을
분리해서 파인튜닝한 KLUE-RoBERTa 모델 두 개(위험도 3-class, 카테고리 7-class)로
분류한다. 결과 화면에는 위험도순 조항 카드와 판정 근거, 그리고 "법률 자문이
아닙니다" 고지가 항상 붙는다.

## 설계 원칙 — 이 프로젝트에서 가장 오래 고민한 부분

**1. 판별은 학습 모델, 설명은 LLM(또는 템플릿).**
LLM에게 "이 조항 위험해?"라고 물으면 그럴듯한 답이 나온다. 하지만 같은 조항에
다른 답이 나올 수 있고, 정량 평가가 어렵고, 왜 그렇게 판단했는지 재현할 수 없다.
위험 판별은 일관성·재현성·평가 가능성이 있는 파인튜닝 분류 모델이 담당하고,
LLM은 판별 결과를 쉽게 설명하는 역할로 제한했다.

**2. False Negative 최소화.**
이 서비스의 최악의 오류는 위험 조항을 "정상"이라고 말하는 것이다. 그래서 평가
1순위 지표를 정확도가 아니라 **danger 클래스 Recall**로 잡았고, 모델 확신이 0.7보다
낮으면 억지로 판정하지 않고 `확인 필요`로 표시한다. 모르는 것을 아는 척하지 않는
것도 설계의 일부다.

**3. 이중 안전망.**
모델 판정과 독립적으로, 라벨링 가이드라인에서 추출한 위험 신호 패턴 14종을 함께
표시한다. 실제 테스트에서 모델이 하향 판정한 위험 조항을 패턴 신호가 잡아낸 사례가
있었다. 모델 하나를 맹신하지 않는 구조가 실제로 일을 했다.

## 기술 스택

- **프론트**: React 18 + Vite, PWA. OCR 실패에 대비해 텍스트 직접 붙여넣기 경로도 뒀다.
- **모델 서버**: FastAPI. 조항 분리 → 분류 → 신호 감지를 담당하고, 프론트와 분리해
  모델 교체·재배포가 독립적이게 했다.
- **모델**: KLUE-RoBERTa-base 파인튜닝 × 2 (위험도, 카테고리)
- **DB/인증**: Supabase (PostgreSQL + RLS) — 히스토리, 조항 피드백 저장

## 배운 점

- "AI 서비스"에서 가장 중요한 결정은 모델 선택이 아니라 **AI에게 무엇을 맡기지
  않을지** 정하는 것이었다. 판별과 설명을 분리한 원칙이 이후 모든 결정의 기준이 됐다.
- 오류에도 등급이 있다. 어떤 오류가 사용자에게 가장 해로운지(위험을 놓치는 것)를
  먼저 정하니, 지표 선택부터 threshold 정책까지 일관되게 따라왔다.
- 도메인의 공백은 기능이 아니라 **시점**에 있었다. 기존 서비스가 못 하는 것은
  분석 기술이 아니라 "계약 당일 눈앞의 문서"라는 상황 대응이었다.
