## 무엇을 하려 했나

블로그를 직접 만들기로 했다. 요구사항은 단순하다.
MDX로 글을 쓰고, 프론트매터가 잘못되면 **빌드가 실패**해야 하고,
글 데이터를 컴포넌트에서 타입 안전하게 다룰 수 있어야 한다.

MDX 파이프라인의 사실상 표준이던 contentlayer는 유지보수가 중단됐다.
대안을 찾다가 Velite를 골랐다.

## 무엇을 했나

Velite는 콘텐츠 스키마를 Zod 기반으로 선언한다. 프론트매터에 `summary`를 빼먹거나
날짜 형식이 틀리면 빌드 타임에 바로 에러가 난다.

```ts
const posts = defineCollection({
  name: "Post",
  pattern: "posts/**/*.mdx",
  schema: s
    .object({
      title: s.string(),
      date: s.isodate(),
      tags: s.array(s.string()),
      summary: s.string().max(160),
      draft: s.boolean().default(false),
      path: s.path(),
      code: s.mdx(),
    })
    .transform((data) => {
      const slug = data.path.replace(/^posts\//, "");
      return { ...data, slug, permalink: `/posts/${slug}` };
    }),
});
```

Next.js와의 연동은 webpack 플러그인으로 처리했다. `next dev`를 켜면 velite가
watch 모드로 함께 돌고, `content/`의 MDX가 `.velite/`의 JSON + 타입 선언으로 변환된다.
앱 코드에서는 이렇게 가져다 쓴다.

```ts
import { posts } from "#site/content";

// posts[0].title — 전부 타입이 잡힌다
```

## 어떤 기술을 어떻게 썼나

- **Velite** — 콘텐츠를 데이터 레이어로 분리. 스키마의 `transform`으로 slug, permalink 같은 파생 필드를 만들었다.
- **s.mdx()** — 본문을 빌드 타임에 컴파일해 문자열로 저장한다. 런타임에는 `new Function`으로 복원만 하므로 MDX 컴파일러가 번들에 들어가지 않는다.
- **경로 별칭** — `#site/content`를 `.velite`로 매핑해, 생성물 위치가 바뀌어도 import 문이 흔들리지 않게 했다.

## 배운 점

- "콘텐츠도 스키마가 있는 데이터"로 취급하면 글이 쌓여도 겁나지 않는다. 잘못된 프론트매터는 배포 전에 죽는다.
- 프레임워크에 붙는 도구를 고를 때는 기능보다 유지보수 상태를 먼저 봐야 한다. contentlayer가 남긴 교훈.
- 이런 인프라 작업은 재미있어서 끝이 없다. "글을 쓰기 위한 블로그"라는 목적을 계속 상기할 것.
