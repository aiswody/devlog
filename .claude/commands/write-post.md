devlog 블로그 글을 작성한다. 인자: $ARGUMENTS (소스 폴더 경로, GitHub 레포명, 또는 주제 설명)

## 절차

1. **소스 파악** — 인자가 로컬 경로면 그 폴더의 README/docs/핵심 코드를 읽는다.
   레포명이면 `gh`로 읽는다. 둘 다 아니면 사용자가 말한 내용이 소스다.
   TIL처럼 짧은 주제면 소스 분석 없이 바로 4번으로.
2. **사실 수집** — 실제 있었던 결정, 수치, 삽질만 모은다. 자료에 없는 것은 지어내지
   않는다.
3. **사용자에게 질문** — 글이 길어질 경우(회고/개발기) 2~3개만 묻는다: 제일 애먹은
   부분, 그 선택을 한 진짜 이유, 결과를 보고 느낀 것. 답을 못 받으면 심경 묘사는
   추정임을 명시하고 마지막에 확인을 요청한다.
4. **작성** — `content/posts/YYYY-MM-slug.mdx`. frontmatter: title, date(오늘),
   tags, summary(≤160자), 필요 시 `type: til` / `project: <slug>` / `draft: true`.
5. **검증·배포** — dev 서버가 떠 있으면 끄고 `pnpm build` → 통과 시
   `content:` prefix로 커밋 → push. 사용자가 초안만 원하면 커밋 전에 멈춘다.

## 문체 (중요 — "AI가 쓴 티" 금지)

- 실제 사건·에러 메시지·수치 기반. 겪지 않은 일을 그럴듯하게 쓰지 않는다.
- 의견과 온도가 있는 문장 ("솔직히 ~라 예상했다", "이 버그는 오랜만이었다").
- 섹션 제목은 내용이 드러나게 ("contentlayer가 죽어 있었다"), 4단 템플릿
  (하려 했나/했나/기술/배운 점)은 뼈대로만 쓰고 기계적으로 반복하지 않는다.
- 충분히 길게 (회고 기준 본문 80줄 이상), 다만 늘리려고 뻔한 문장을 넣지 않는다.
- 배운 점은 구체적으로 — "협업의 중요성을 배웠다" 같은 문장 금지.

## 제약

- 팀 프로젝트: 팀원 실명 금지 ("5인 팀"), 본인 담당 파트를 명확히.
- 의료 데이터(MIMIC 등): 환자 단위 데이터/출력 인용 금지.
- 색·폰트·컴포넌트 등 블로그 코드 수정은 이 명령어 범위 밖 — 글만 쓴다.
